## Quick Start

### Prerequisites

- Python 3.10+ (project uses Streamlit + Flask)
- PostgreSQL running (required for mentee CRUD)
- A Gemini API key in `.env` (for AI features)

---

### 1. Configure your environment variables

Copy the example file and fill in your credentials:

```bash
cp .env.example .env
```

Then open `.env` and fill in the following values:

```env
# Gemini AI — get your key at aistudio.google.com
GEMINI_API_KEY="your_gemini_api_key_here"

# PostgreSQL connection
DB_HOST="localhost"
DB_PORT="5432"
DB_NAME="your_database_name"
DB_USER="your_database_user"
DB_PASSWORD="your_database_password"
```

> Never commit `.env` to version control. Make sure it's listed in your `.gitignore`.

---

### 2. Running the App

Start the backend and frontend with a single command:

```bash
python main.py
```

This starts both the Flask backend and the Streamlit frontend. Once running, open the URL shown in your terminal.

---

### 3. Running Tests

Run the full mentee test suite with verbose output:

```bash
pytest test_mentees.py -v
```

**Did you use an AI assistant (Copilot, ChatGPT, etc.)?** Yes

**How did you verify the suggestions?**
I didn't blindly copy-paste, I reviewed every suggestion before using it. I ran the code locally, checked edge cases, and used pytest to confirm behaviour matched expectations. If a test failed or output looked wrong, I debugged it. Fallback behavior when Gemini fails/unavailable (rule-based keyword extraction). Verified robustness with automated tests.

**Give one example of a suggestion you rejected or changed:**
I initially considered using a heavier model for the voice/pivot flow (based on suggestion), but kept the implementation focused on Gemini 2.5 Flash-style behavior (fast and appropriate for the task) and prioritized reliability via explicit fallbacks and input validation instead of using a larger model which would take time. I also used python library to parse cv which was faster than using gemini api call.

## Tradeoffs & Prioritization

**What did you cut to stay within the 4–6 hour limit?**
Cut "extra" features outside the required core flow (like richer frontend streaming and UX polishing). Added more features that are useful for the user.

---

**What would you build next if you had more time?**

1. **Production-stable real-time audio** — The current WebSocket connection works but isn't resilient — I'd add exponential backoff reconnect logic, smarter audio chunk buffering, and proper voice activity detection (VAD) for cleaner turn-taking. The goal is a session that survives network hiccups without the user noticing.

2. **Broader test coverage + CI pipeline** — Right now tests cover the mentee CRUD layer. I'd extend coverage to the AI endpoints using mocked Gemini responses — testing prompt formatting, response parsing, and edge cases like empty or malformed model output. Then wire it all into a GitHub Actions CI pipeline so every push is validated automatically.

## Known Limitations

**Real-time audio streaming** — Native audio streaming via Gemini Live requires runtime support and model availability. It may not function in restricted networks, limited environments, or where Gemini Live access is unavailable — the app falls back gracefully in these cases.

**Roadmap output variability** — Roadmap suggestions are generated by the AI model and shaped by the quality and coverage of the provided CSV datasets. Outputs may vary across runs — treat them as intelligent starting points rather than fixed recommendations.

**Not optimised for production scale** — This prototype prioritises correctness and graceful fallback behaviour over performance. It is not designed for production-level latency or high-concurrency workloads — scaling optimisations are planned for future iterations.
